## 后续3：首次宽度修复首次实机仍无效 → 二次修复

## 进入 Agent 页按需重启判据重构（用户再次反馈 bug）
- 用户的真实诉求（纠正上一版逻辑）：
  1. ctx 没变 + 模型没重启过 → 哪怕切页面也**不应自动退出** admAgent。
  2. admAgent 进程已退出（崩溃）→ 重新进入 Agent 页应**自动重新拉起**。
  3. 隐含：模型被重启过也应触发重启（原逻辑未覆盖）。
- 上一版 bug：只比较 `currentCtxSize`，且用前端 `terminalStarted` 标记代替"进程是否真活着"——进程崩溃后标记未复位，误判为"仍在运行"导致不重启。
- 重构（以"后端权威状态"为准）：
  - 后端 `agent.rs` 新增 `get_agent_status`：`try_wait()` 查 PTY 子进程存活（`running`）+ 返回 `model_generation`。
  - `app_state.rs` 加 `model_generation: Mutex<u64>`；`model_list.rs` 的 `start_model` 成功时 `bump_model_generation()`。
  - 前端 `handleEnterAgent()`：进入页拉 `get_agent_status`+`get_agent_ctx_size`，仅当"进程存活 && ctx 未变 && 模型代次未变"才保持原状；否则一律重启（`start_agent_terminal` 内部先清理旧会话再拉起）。`startAgentNow` 记录 `currentCtxSize`+`currentModelGen`，并在每次启动前 `term.reset()`。
- `cargo check` 通过。
- 经验：常驻进程"是否还在运行"必须用 `try_wait()` 等真实探测，不能信前端一次性标记或"上次成功"记忆。

## 重启判据最终口径：以"模型是否重启"为准（用户再次纠正）

## 修复：Agent 终端 Ctrl+C / Ctrl+V 复制粘贴
- 现象：用户反馈 agent 页面无法用 Ctrl+C / Ctrl+V 复制粘贴，想要"像真实终端一样"的效果。

- 修复（纯前端 `src/agent.html` + `src/index.html`，无需 cargo check）：

  - iframe 内 `navigator.clipboard.readText` 受限时，通过 `agent-read-clipboard` 事件委托父窗口 `index.html` 代理读取，回传 `agent-clipboard-result`。读取失败终端内提示。
  - `doc/dev_doc.md` 13.2 节补充复制/粘贴行为说明。
