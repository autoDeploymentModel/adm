## 后续2：Agent 页首次进入终端宽度不适配
- 现象：1440×800 窗口首次进 Agent 页，xterm 终端宽度没适配（退回默认列数/右侧留白），二次进入才正常。
- 根因：`agent.html` 的 `initTerminal` 在 `term.open()` 后同步 `fitAddon.fit()`，iframe 刚从 display:none 切可见、布局未完成，fit 算错列数；且后端 PTY 用固定 120×30 创建，首次启动后未把前端真实列数同步回去。
- 修复（`src/agent.html`，纯前端）：
  1. `initTerminal` 的首次 fit 改用 `requestAnimationFrame` 延迟一帧执行，并 fit 后 `agent_terminal_resize` 同步后端。
  2. `agent-terminal-ready` 事件处理里增加 `fit()` + `agent_terminal_resize`（此时 PTY 已建，resize 生效），纠正后端初始 120×30。
- 无需 cargo check（仅前端 HTML/JS）。

## 后续3：首次宽度修复首次实机仍无效 → 二次修复
- 用户反馈 Windows 实机首次进入仍没适应。根因不是 rAF 时机，而是**后端 `agent-terminal-ready` 事件早于 agent.html 注册 message 监听被丢失**（首次要解析 289KB xterm.js，加载慢于后端 emit ready），导致依赖该事件的 fit 从未执行；只靠布局未稳时的那次 fit 算错列数。
- 二次修复（`src/agent.html`）：
  1. `initTerminal` 用 `ResizeObserver` 监听 `#terminal-wrap` 兜底适配；`observe` 后立即回调一次真实尺寸（首次进入本页已加载、容器已可见），窗口缩放也自动适配。PTY 未建时 resize 为 no-op。
  2. `startAgentNow` 在 `invoke("start_agent_terminal")` 成功后（确定 PTY 已建）主动 `fit()` + `agent_terminal_resize` 同步后端；不依赖任何一次性事件。
- 结论：前端适配应优先用 ResizeObserver + 在确定 PTY 已建后的主动 resize，避免依赖可能被 iframe 加载时序错过的一次性 Tauri 事件。

## 上下文变化自动重启 admAgent（用户复现的 bug）
- 复现：60K 进 agent → 设置改 128K 重启模型 → 回 agent 页仍 60K。
- 根因：从设置回 agent 页，`agent-frame` 仅隐藏再显示，`terminalStarted=true`，`agent-resize` 不重启终端；已运行 admAgent 读启动时的旧 ctx，admAgent.json 虽被更新但进程不重读。
- 修复：`agent.rs` 新增命令 `get_agent_ctx_size`（返回 settings.launch_params.ctx_size）；`lib.rs` 注册。`agent.html` 加 `currentCtxSize` 状态 + `handleEnterAgent()`：每次进入（`agent-resize`）比较当前 ctx 与 currentCtxSize，不一致则先 `stop_agent_terminal` 再 `startTerminal`，由 `ensure_adm_agent_config` 用新 ctx 重生配置并重启 admAgent；未变则不重启。`startAgentNow` 成功后记录 currentCtxSize。`cargo check` 通过。
- 经验：iframe 常驻终端场景下，"进入页面"应作为上下文/配置变更的检查点，不能只依赖首次启动。

## 进入 Agent 页按需重启判据重构（用户再次反馈 bug）
- 用户的真实诉求（纠正上一版逻辑）：
  1. ctx 没变 + 模型没重启过 → 哪怕切页面也**不应自动退出** admAgent。
  2. admAgent 进程已退出（崩溃）→ 重新进入 Agent 页应**自动重新拉起**。
  3. 隐含：模型被重启过也应触发重启（原逻辑未覆盖）。
- 上一版 bug：只比较 `currentCtxSize`，且用前端 `terminalStarted` 标记代替"进程是否真活着"——进程崩溃后标记未复位，误判为"仍在运行"导致不重启。
- 重构（以"后端权威状态"为准）：
  - 后端 `agent.rs` 新增 `get_agent_status`：`try_wait()` 查 PTY 子进程存活（`running`）+ 返回 `model_generation`。
  - `app_state.rs` 加 `model_generation: Mutex<u64>`；`model_list.rs` 的 `start_model` 成功时 `bump_model_generation()`。
  - 前端 `handleEnterAgent()`：进入页拉 `get_agent_status`+`get_agent_ctx_size`，仅当"进程存活 && ctx 未变 && 模型代次未变"才保持原状；否则一律重启（`start_agent_terminal` 内部先清理旧会话再拉起）。`startAgentNow` 记录 `currentCtxSize`+`currentModelGen`，并在每次启动前 `term.reset()`。
- `cargo check` 通过。
- 经验：常驻进程"是否还在运行"必须用 `try_wait()` 等真实探测，不能信前端一次性标记或"上次成功"记忆。

## admAgent TUI 右边栏上下文统计显示错位（用户澄清：非宽度/内容，是 admAgent 自身 TUI）
- 现象：嵌入终端里 admAgent 右边栏"上下文实时统计"显示不正常，独立 powershell 正常。
- 根因：后端 `start_agent_terminal` 创建 PTY 写死 120×30 列；admAgent 启动时读 PTY 宽度(120)布局 TUI，而 xterm 实际显示更宽 → 错位。事后 `agent_terminal_resize` 于启动后才调，已无意义。
- 修复：`start_agent_terminal` 加 `rows`/`cols` 参数（前端真实尺寸）创建 PTY；`agent.html` 的 `startAgentNow` 先 `fitAddon.fit()` 取真实尺寸再 `invoke` 创建 PTY；shell 命令加 `TERM=xterm-256color`。
- `cargo check` 通过。经验：嵌入式终端里"TUI 布局错位"几乎都是 PTY 初始尺寸 ≠ 显示尺寸导致，务必在创建 PTY 时传入真实尺寸，而不是事后 resize。

## 重启判据最终口径：以"模型是否重启"为准（用户再次纠正）
- 用户明确：进入 Agent 页是否重启 admAgent，**唯一标准是"模型是否重启过"**，不再比较上下文大小。
  - 模型重启过（代次变化）→ 再次进入必须重启；
  - 即便只改了上下文配置但没重启模型 → **不重启**（保留原终端）；
  - 进程已退出（崩溃）→ 仍要重新拉起（恢复）。
- 改动：移除前端 `currentCtxSize` 状态与 `get_agent_ctx_size` 调用；`handleEnterAgent()` 改为只拉 `get_agent_status`，仅当"进程已退出 || 模型代次变化"时重启；后端删除无用的 `get_agent_ctx_size` 命令（`agent.rs`+`lib.rs`）。`cargo check` 通过。
- 结论：admAgent 的 restart 触发条件 = 进程死亡 OR 模型代次变化；上下文配置变更本身不是触发条件。
