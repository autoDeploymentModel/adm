# 2026-07-16

## 修复：agent.html 鼠标滚轮失效
- 根因：agent.html 内两段代码刻意禁用滚轮 —— CSS `.xterm-viewport { overflow-y: hidden }` 关掉视口滚动；JS 在捕获阶段 `addEventListener("wheel", preventDefault+stopPropagation)` 把事件掐断，导致 xterm 无法把滚轮转发给已启用鼠标追踪的 admAgent TUI。
- 改动：`src/agent.html`
  - CSS `overflow-y: hidden` → `auto`（保留滚动条隐藏外观）。
  - 删除 JS 的 wheel 拦截监听器，仅留注释说明不要拦截。
- 修复后：TUI 启用鼠标追踪时滚轮转发给应用；未启用时滚动 xterm 回滚区。前端无构建步骤，重新 `pnpm tauri dev` 或重启应用即生效。

## 二次修复：滚轮仍失效（兜底转发）
- 用户反馈上一版仍无效。深挖 `src/vendor/xterm/xterm.js` 发现：xterm 仅在「鼠标追踪协议带 wheel 位（DECSET 1000/1002/1003）」时才会绑定 wheel 监听并把滚轮编码成鼠标报文转发给 PTY；若 admAgent TUI 实际未走标准追踪（或该路未生效），滚轮就到不了应用。
- 改动：`src/agent.html`（同在 `initTerminal` 内）
  - 把之前删除的「拦截器」换成**正向转发器**：在捕获阶段拦截 wheel，`preventDefault + stopPropagation`（仅对 wheel，不影响点击/拖选），把滚轮换算成 **SGR 鼠标报文** `\x1b[<(64+action+mods);col;rowM`（action 0=上/1=下），经 `agent_terminal_input` 直接写入 PTY。
  - deltaMode 行/页模式统一换算成像素累加；跨过 ~24px 阈值才发一次，避免触控板微事件刷屏。
  - stopPropagation 阻止 xterm 自身 wheel 监听重复触发，保证只发一次。
- 前提：admAgent 用 SGR(1006) 编码读鼠标（crossterm/ratatui 默认）。若其用普通 1000 编码，需改报文格式。

## 三次修复 + 关键排查结论（07-18 补充）
- 用户仍反馈滚轮不生效。已验证：后端 `agent_terminal_input` 把 base64 解码后 `write_all` 到 PTY master（与键盘同链路，键盘能打字即证明通）；`TERM=xterm-256color`；admAgent 二进制字符串确含 `?1006h`/`?1003h` 与大量 wheel/scroll → 用 SGR + any-event 鼠标，格式与我发的 SGR 完全一致。代码逻辑应正确。
- **真正卡点（高概率）：`index.html` 的 `agentFrameLoaded` 守卫让 agent iframe 只加载一次，`src` 不重设；反复进出 Agent 页不会重载 iframe 文档。** 因此改了 `src/agent.html` 后，只要不重启 `pnpm tauri dev`/应用，iframe 里仍是旧 JS —— 这就是"改了多次都不生效"的主因。
- 本轮改动（`src/agent.html`）：把 wheel 监听器从 `#terminal` 改挂到 **`window` 捕获阶段**（最稳，能确认事件是否进到 iframe）；并加**临时调试浮标**（右下角闪「wheel ▼/▲ → PTY」）用于确认监听器确实在跑。
- 给用户的关键指引：改完后必须**重启 `pnpm tauri dev`（或整个应用）**让 agent iframe 重新加载新代码；若看到浮标但 TUI 不滚 → 代码在跑、是 admAgent 侧问题（需进一步抓 PTY 日志）；若看不到浮标 → 事件没进 iframe 或仍是旧构建。
- 待确认后移除调试浮标。
